# MVP0-003 — Indexing

## Goal

Организовать назначение ролей и привязку прав доступа к пользователю так, чтобы каждый запрос однозначно определял, что разрешено.

> В контексте Reps «индексация» — построение связей пользователь ↔ роль и подготовка данных к проверкам доступа.

## Requirements

### Роли

- Три роли: `admin`, `coach`, `athlete`
- Роль назначается при регистрации (`coach` / `athlete`) или через seed (`admin`)
- Роль неизменяема пользователем самостоятельно в MVP 0

### Права доступа

- `admin` — полный доступ к списку пользователей
- `coach` — личный кабинет тренера; нет доступа к списку всех пользователей
- `athlete` — личный кабинет спортсмена; нет доступа к чужим данным
- Middleware / guard проверяет роль на каждом защищённом маршруте

### БД и целостность

- Индекс на `users.email` (уникальный)
- Индекс на `users.role`
- Роль хранится как enum / constrained value

## Acceptance Criteria

- [x] Каждый пользователь имеет ровно одну роль
- [x] Защищённые маршруты отклоняют запросы без нужной роли (403)
- [x] Неаутентифицированный доступ к защищённым маршрутам отклоняется (401)
- [x] Проверка роли вынесена в переиспользуемый механизм (guard / dependency)
- [x] Unit-тесты на матрицу ролей и доступа
- [x] Integration-тесты на типовые сценарии авторизации
