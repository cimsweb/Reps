# MVP2-003 — Indexing

## Goal

Обеспечить целостность связей планов и отчётов с парой тренер ↔ спортсмен и подготовить БД к выборкам по календарным периодам (неделя / месяц).

> В контексте Reps «индексация» — инварианты принадлежности данных и индексы для запросов по датам.

## Requirements

### Связи и инварианты

- `TrainingPlan` и `PlannedWorkout` принадлежат паре coach + athlete (связь из MVP 1)
- `WorkoutCompletionReport` принадлежит спортсмену и ссылается на `PlannedWorkout`
- Тренер не видит планы и отчёты несвязанных спортсменов
- Спортсмен не видит чужие планы и отчёты
- Удаление пользователя каскадно удаляет его планы и отчёты (или явное правило в миграции)

### Индексы БД

- `planned_workouts (athlete_id, planned_date)` — выборка плана на неделю / месяц
- `planned_workouts (coach_id, planned_date)` — списки тренера
- `workout_completion_reports (athlete_id, created_at)` — итоги за период
- `workout_completion_reports (planned_workout_id)` UNIQUE — один отчёт на тренировку

### Application layer

- `TrainingAccessGuard` (или расширение `CoachingAccessGuard`) — проверка доступа к планам и отчётам
- Атомарное сохранение плана на неделю (все тренировки в одной транзакции)

## Acceptance Criteria

- [x] План и отчёты однозначно привязаны к coach ↔ athlete
- [x] Тренер получает доступ только к данным своих спортсменов
- [x] Спортсмен видит только свои планы и отчёты
- [x] Индексы БД созданы и задокументированы
- [x] Unit-тесты на правила принадлежности
- [x] Integration-тесты на целостность при создании недельного плана
