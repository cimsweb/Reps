# MVP1-003 — Indexing

## Goal

Организовать связи и агрегацию данных между тренером и спортсменом так, чтобы информация была консистентной, быстро доступна для просмотра и не дублировалась.

> В контексте Reps «индексация» — это не поисковый индекс и не эмбеддинги, а построение связей и подготовка данных к запросам тренера.

## Requirements

### Связи

- При принятии приглашения создаётся устойчивая связь тренер ↔ спортсмен
- У одного спортсмена может быть несколько тренеров (если не ограничено доменом — зафиксировать в задаче при реализации)
- Разрыв связи — вне scope MVP 1, unless explicitly needed

### Агрегация данных

- Профиль спортсмена привязан к `athlete_id`
- Рекорды группируются по спортсмену и типу (дистанция / упражнение)
- Обратная связь привязана к спортсмену и доступна связанному тренеру
- Ссылки Garmin хранятся в составе `WorkoutFeedback`

### Производительность и целостность

- Индексы БД на `coach_id`, `athlete_id`, `invitation.email`
- Каскадные правила согласованы с доменом (что удаляется вместе со спортсменом)
- Нет orphan-записей после принятия/отклонения приглашения

## Acceptance Criteria

- [x] Связь тренер ↔ спортсмен создаётся атомарно при принятии приглашения
- [x] Профиль, рекорды и feedback однозначно принадлежат спортсмену
- [x] Тренер через связь получает доступ только к данным своих спортсменов
- [x] Запрос списка спортсменов тренера выполняется без N+1
- [x] Индексы БД созданы и задокументированы
- [x] Unit-тесты на правила принадлежности данных
- [x] Integration-тесты на целостность связей
