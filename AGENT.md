# Project Agent Instructions

> Читать этот файл первым. Детали каждого релиза — в `tasks/`.

## Project Goal

Reps — платформа для тренера и спортсмена, где план превращается в прогресс.

Единое пространство для персональных тренеров по бегу, трейлу и OCR и их атлетов: план тренировок, отчёт о выполнении и обратная связь в одном месте. ИИ усиливает тренера — помогает выстраивать программу и анализировать нагрузку, не заменяя его.

## MVP Scope

Релизы идут строго по порядку. Не начинать следующий MVP, пока не закрыт предыдущий.

| MVP   | Обзор                            | Задачи              |
| ----- | -------------------------------- | ------------------- |
| MVP 0 | [tasks/mvp-0.md](tasks/mvp-0.md) | MVP0-001 … MVP0-005 |
| MVP 1 | [tasks/mvp-1.md](tasks/mvp-1.md) | MVP1-001 … MVP1-005 |
| MVP 2 | [tasks/mvp-2.md](tasks/mvp-2.md) | MVP2-001 … MVP2-005 |
| MVP 3 | [tasks/mvp-3.md](tasks/mvp-3.md) | MVP3-001 … MVP3-005 |

Каждый MVP декомпозирован на 5 задач: **Project Setup → Data Ingestion → Indexing → Search → UI**. Детали — в файлах `tasks/MVP*-00x-*.md`.

### Входит в текущий MVP (MVP 0)

- Регистрация и вход
- Три роли: администратор, тренер, спортсмен
- Личный кабинет под каждую роль
- Администратор видит всех пользователей

**Цель:** рабочая система аутентификации с разграничением прав.

### Не входит в текущий MVP

Всё из MVP 1–3 и продуктовые возможности вне плана релизов:

- Приглашения спортсменов, профили, рекорды, обратная связь по тренировкам (MVP 1)
- ИИ, таблицы результатов, графики динамики (MVP 2)
- Garmin API, автоматический импорт активностей (MVP 3)
- Мобильные приложения, платежи, социальные функции, клубы как отдельная сущность

## Architecture Principles

- Простота — решение должно быть понятным без лишних слоёв
- Минимум зависимостей — новая библиотека только при явной необходимости
- Чистые интерфейсы — transport, application и domain разделены; зависимости направлены внутрь
- Один use case — одна ответственность
- YAGNI — не строить инфраструктуру под будущие MVP заранее

Целевая структура каталогов описана в `.cursor/rules/Python-Development-Rules.mdc` (раздел DDD structure).

## Development Rules

- Не добавлять новые библиотеки без согласования
- Не менять публичный API без обновления документации
- Следовать структуре каталогов из правил проекта
- Реализовывать только scope текущего MVP из `tasks/`
- Не переносить фичи из следующего MVP «на будущее» в текущий код
- Перед коммитом: `ruff check .`, `ruff format .`, `mypy src/`, `pytest` (backend); `npm run lint`, `npm run format:check` (frontend)

## Coding Standards

Стандарты кода, DDD-структура, тестирование, форматирование и именование — в `.cursor/rules/Python-Development-Rules.mdc`. Следовать им без дублирования здесь.

Проектные требования:

- Python >= 3.13 (`pyproject.toml`)
- Type hints на публичных функциях и методах
- Docstrings на публичном API
- Новое поведение покрывается тестами

## Workflow

1. **Тесты** — написать или обновить тесты под задачу
2. **Реализация** — минимальный код, закрывающий тесты и scope MVP
3. **Документация** — обновить README, API-описание или task-файл при изменении контрактов

При работе над фичей открывать соответствующий файл в `tasks/` и сверяться с чеклистом.
